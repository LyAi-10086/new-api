# 管理端数据统计重构规划

## 1. 背景与目标

当前管理端已经有 `/data-statistics` 数据统计页，并且后端已经提供第一版接口：

- `GET /api/data-statistics/summary`
- `GET /api/data-statistics/trends`
- `GET /api/data-statistics/rankings`
- `GET /api/data-statistics/filters`

第一版已经能展示充值、消耗、登录、用户、模型、分组、渠道等只读统计，但页面仍偏“区间汇总 + 趋势表格”，管理员需要自己从表格里判断趋势，无法直观看到用户增长、收入变化、消耗波动、错误率异常和渠道贡献。

本次重构目标是把该页面升级为真正的“运营数据驾驶舱”：

- 一眼看清平台总盘：总用户、余额、累计消耗、区间收入、区间请求、错误率。
- 一眼看清历史趋势：哪天新增多少用户、总用户增长到多少、哪天充值/消耗/错误率波动。
- 一眼看清贡献结构：哪些模型、分组、渠道、用户贡献最多。
- 一眼看清异常风险：错误率升高、耗时升高、余额异常增长、消耗异常降低或反向增长。
- 尽量复用当前接口和表结构，第一版不引入复杂预聚合系统，降低后续合并上游的冲突。

## 2. 产品层面规划

### 2.1 页面定位

页面定位从“数据列表”调整为“运营看板”：

- 面向超级管理员和运营人员。
- 用于日常巡检、运营复盘、渠道排障、模型成本观察、用户增长分析和异常账户复核。
- 只读展示，不在该页面直接修改用户余额、渠道、模型、价格等核心配置。

### 2.2 信息架构

建议页面分为 5 个区域：

1. 筛选区
2. 总览区
3. 趋势区
4. 结构区
5. 排行与异常区

页面结构：

```text
│   ├── 渠道
│   ├── 支付提供商
│   └── 用户 ID

│   ├── 总用户数

│   ├── 登录用户

├── 趋势区
```

### 2.3 总览区

总览卡片需要从“区间数字”升级为“当前值 + 环比变化 + 辅助解释”。

推荐卡片：

| 卡片 | 主值 | 辅助值 | 环比 |
| --- | --- | --- | --- |
| 总用户数 | 当前用户总数 | 区间新增用户 | 较上一周期新增多少 |
| 新增用户 | 区间新增用户 | 日均新增 | 较上一周期变化 |
| 活跃用户 | 区间内发起请求的用户 | 登录用户数 | 较上一周期变化 |
| 登录 | 登录次数 | 登录用户数 | 较上一周期变化 |
| 充值金额 | 区间成功充值金额 | 充值次数/充值额度 | 较上一周期变化 |
| 消耗金额 | 区间消耗额度折算金额 | 请求数 | 较上一周期变化 |
| 请求数 | 成功请求数 | 流式占比 | 较上一周期变化 |
| Token 数 | 总 Token | 输入/输出 Token | 较上一周期变化 |
| 错误率 | 错误率 | 错误数 | 较上一周期变化 |
| 平均耗时 | 平均响应耗时 | P95 可后续加入 | 较上一周期变化 |
| 当前总余额 | 全站用户当前余额 | 高余额用户数量 | 较上一周期快照变化可后置 |
| 累计已消耗 | 全站累计 used_quota | 累计请求数 | 较上一周期快照变化可后置 |

说明：

- 区间类指标可以实时从日志和充值记录聚合。
- 当前总余额、累计已消耗属于当前状态快照，第一版直接从 `users` 表聚合。
- 当前总余额的历史趋势需要每日快照，第一版可以先只展示当前值，第二版再加历史。

### 2.4 趋势区

趋势区是本次重构的重点。当前趋势表格保留，但应该降级为“明细表”，图表成为主展示。

#### 用户趋势

展示内容：

- 柱状图：每日新增用户。
- 折线图：累计总用户。
- 折线图：活跃用户。
- 折线图：登录用户。

管理员可以直观看到：

- 哪天新增用户变多。
- 当前总用户增长到多少。
- 新增用户和活跃用户是否匹配。
- 注册增长后是否真的带来使用。

#### 收入趋势

展示内容：

- 柱状图：每日充值金额。
- 折线图：充值次数。
- 可选拆分：支付提供商占比。

管理员可以直观看到：

- 哪天充值高峰。
- 充值金额和充值次数是否同步。
- 某个支付渠道是否突然没有收入。

#### 消耗趋势

展示内容：

- 折线图：消耗额度/折算金额。
- 柱状图：请求数。
- 折线图：总 Token。
- 可选拆分：输入 Token、输出 Token。

管理员可以直观看到：

- 收入和消耗是否匹配。
- 用户请求是否突然降低。
- 某个模型或分组是否消耗异常。

#### 质量趋势

展示内容：

- 折线图：错误率。
- 柱状图：错误数。
- 折线图：平均耗时。
- 柱状图：流式请求数或流式占比。

管理员可以直观看到：

- 哪天错误率升高。
- 哪天耗时升高。
- 渠道告警、模型可用性和实际请求质量是否对应。

### 2.5 结构区

结构区用于回答“钱和消耗从哪里来”。

推荐图表：

- 模型消耗占比：Top 模型横向条形图。
- 分组消耗占比：Top 分组横向条形图。
- 渠道消耗占比：Top 渠道横向条形图。
- 支付渠道占比：支付提供商充值占比。

第一版不建议使用复杂饼图。横向条形图更适合长模型名、长分组名和渠道名，信息密度更高。

### 2.6 排行与异常区

保留当前排行榜，但增强两个方向：

- 排行更全。
- 异常更明确。

推荐排行：

- 模型消耗排行。
- 分组消耗排行。
- 渠道消耗排行。
- 用户消耗排行。
- 用户充值排行。
- 用户余额排行。
- 错误模型排行。
- 错误渠道排行。
- 高错误率模型排行。
- 高错误率渠道排行。

推荐异常：

- 余额异常增长用户。
- 消耗为负或结算异常记录。
- 请求数突然下降的模型。
- 错误率突然升高的模型。
- 错误率突然升高的渠道。

异常区可以先做只读提示，不直接提供处置按钮，避免把统计页变成操作后台。

## 3. 管理员层面规划

### 3.1 日常巡检场景

管理员每天打开页面后，应该能快速回答：

- 昨天新增多少用户。
- 当前总用户多少。
- 昨天充值多少。
- 昨天消耗多少。
- 错误率是否异常。
- 哪些模型用得最多。
- 哪些渠道最不稳定。

推荐默认视图：

- 时间范围：近 7 天。
- 粒度：天。
- 展示全站数据。
- 总览区展示环比上一周期。

### 3.2 运营复盘场景

管理员做周报或月报时，应该能回答：

- 近 30 天用户增长曲线。
- 近 30 天充值曲线。
- 近 30 天消耗曲线。
- 哪些模型带来主要消耗。
- 哪些分组是主要用户群。
- 充值用户和消耗用户是否一致。

推荐功能：

- 快捷时间：今日、昨日、近 7 天、近 30 天、近 90 天。
- 粒度：小时、天、周、月。
- 过滤模型、分组、渠道、支付提供商。
- 图表下面保留明细表，方便复制数据。

### 3.3 渠道排障场景

当用户反馈某些模型不可用或速度慢，管理员可以：

- 按渠道筛选。
- 查看该渠道错误率趋势。
- 查看该渠道平均耗时趋势。
- 查看该渠道消耗和请求是否下降。
- 对照 `/channel-alerts` 和 `/model-availability` 页面进一步定位。

### 3.4 安全复核场景

针对计费漏洞、异常余额增长、负数扣费等风险，管理员需要快速定位：

- 哪些用户余额异常增长。
- 哪些用户 used_quota 异常下降或出现负数。
- 哪些日志的 quota 为负数。
- 哪些模型或渠道在某个时间段出现异常扣费。
- 哪些用户在短时间内产生极大请求量或极大 token 数。

第一版建议在数据统计页增加只读“异常观察”模块，后续再做专门的风控审计页。

### 3.5 权限与可见性

- 页面继续只允许 Root 管理员访问。
- 不对普通管理员或普通用户开放。
- 不展示 prompt 原文、请求体、密钥、令牌明文。
- 只展示聚合指标、排行、用户 ID、用户名等管理侧已有信息。

### 4.2 数据来源

核心数据来源：

| 数据 | 主要来源 | 说明 |
| --- | --- | --- |
| 请求数 | `logs` | `type = consume` |
| 消耗额度 | `logs.quota` | 需要注意负数异常 |
| Token | `logs.prompt_tokens`、`logs.completion_tokens` | 输入、输出、总量 |
| 错误 | `logs` | `type = error` |
| 登录 | `logs` | `type = login` |
| 用户 | `users` | 注册、余额、累计消耗 |
| 充值 | `topups` | 成功充值金额、额度、支付提供商 |
| 渠道 | `channels` + `logs.channel_id` | 渠道排行、渠道质量 |
| 模型 | `logs.model_name` 或映射后的展示字段 | 要和使用日志模型展示逻辑保持一致 |
| 分组 | `logs.group` 或兼容字段 | 分组排行、分组筛选 |
| 渠道告警 | `channel_alerts` 相关表 | 后续用于异常联动 |
| 模型可用性 | 模型观测聚合 | 后续用于质量联动 |
| 邀请返利 | 返佣记录表 | 后续可加入运营收入/成本分析 |
| 敏感词违规 | `sensitive_violations` | 后续可加入风控趋势 |

### 4.3 接口设计

第一阶段尽量扩展现有接口，减少新路由。

#### 4.3.1 总览接口

保留：

```text
GET /api/data-statistics/summary
```

建议返回结构升级：

```json
{
  "filter": {},
  "summary": {
    "current": {},
    "previous": {},
    "delta": {},
    "delta_rate": {}
  }
}
```

如果为了兼容当前前端，也可以先保持原字段不变，新增：

```json
{
  "summary": {},
  "previous_summary": {},
  "delta": {},
  "delta_rate": {}
}
```

这样旧字段不破坏，新前端可以使用新增字段。

#### 4.3.2 趋势接口

保留：

```text
GET /api/data-statistics/trends
```

新增字段：

```json
{
  "bucket": 1783353600,
  "registered_users": 12,
  "total_users": 1024,
  "active_users": 88,
  "login_users": 67,
  "topup_money": 123.45,
  "topup_amount": 1000000,
  "topup_count": 9,
  "consume_quota": 888888,
  "request_count": 3000,
  "prompt_tokens": 100000,
  "completion_tokens": 120000,
  "total_tokens": 220000,
  "error_count": 30,
  "error_rate": 0.01,
  "avg_use_time": 1.2,
  "stream_count": 1800,
  "stream_ratio": 0.6
```

重点新增：

- `total_users`：截至当前 bucket 的累计总用户。
- `topup_count`：充值次数。
- `success_request_count` 可选：如果需要和错误请求区分更清楚。

#### 4.3.3 结构接口

第一版可复用：

```text

```

后续如果排行榜越来越重，再拆：

```text
GET /api/data-statistics/breakdown

```

建议第一版先在 `rankings` 中新增：

- `error_models`
- `error_channels`
- `high_error_rate_models`
- `high_error_rate_channels`
- `payment_providers`

### 4.4 查询策略

#### 实时聚合优先

第一版继续使用实时聚合：

- 日志表按时间范围聚合。
- 充值表按完成时间聚合。
- 用户表按注册时间聚合。
- 用户总量通过 `created_at <= bucket_end` 计算。

这样不需要迁移新表，最小化和上游冲突。

#### 快照表后置

当数据量变大后，再新增每日快照表：

```text
admin_data_statistics_daily_snapshots
```

建议字段：

```text
id
date
group
model_name
channel_id
total_users
new_users
active_users
login_users
topup_money
topup_count
consume_quota
request_count
total_tokens
error_count
avg_use_time
created_at
updated_at
```

第二阶段再考虑：

- 每日定时生成快照。
- 管理员手动重算某天快照。
- 近 180 天优先读快照，不足部分实时补齐。

### 4.5 时间范围限制

为了避免统计页拖垮生产数据库：

- 默认范围：近 7 天。
- 最大范围：180 天。
- 小时粒度：最多 7 天。
- 天粒度：最多 180 天。
- 周/月粒度：第二阶段加入，可支持更长范围。

如果用户选择超过限制，后端返回明确错误：

```text
小时粒度统计时间范围不能超过 7 天
统计时间范围不能超过 180 天
```

### 4.6 环比计算

环比使用同等长度的上一周期。

示例：

- 当前范围：2026-07-01 到 2026-07-07
- 上一周期：2026-06-24 到 2026-06-30

返回：

- 当前值
- 上期值
- 差值
- 变化率

变化率规则：

- 上期为 0，当前也为 0：变化率为 0。
- 上期为 0，当前大于 0：变化率返回 `null`，前端显示“新增”。
- 当前小于上期：显示下降。

### 4.7 安全与边界

统计页必须只读：

- 不修改余额。
- 不修改充值记录。
- 不修改日志。
- 不修改渠道状态。
- 不修改用户状态。

敏感数据处理：

- 不展示 prompt 原文。
- 不展示完整请求体。
- 不展示令牌明文。
- 不展示 API Key。
- 不在服务日志输出大字段统计结果。

异常数值处理：

- 后端聚合时要识别负数 `quota`。
- 负数 quota 不应该被静默合并成正常消耗。
- 可额外返回 `negative_quota_count` 和 `negative_quota_sum`，用于安全复核。
- 对超大数值聚合要使用安全类型，避免统计页再次出现整数溢出风险。

新增粒度：

- 小时
- 天
- 周
- 月

第一版可先支持小时和天；周/月可以在后端具备 bucket 后补齐。

## 6. 分阶段实施计划

## 7. 验收清单

### 产品验收

- 管理员打开页面后，3 秒内能看懂全站当天/近 7 天状态。
- 能直接看出哪天新增用户最多。
- 能直接看出当前总用户数。
- 能直接看出哪天充值最多。
- 能直接看出哪天消耗最多。
- 能直接看出错误率是否异常。
- 能直接看出主要消耗来自哪些模型、分组和渠道。

### 技术验收

- `/api/data-statistics/summary` 保持兼容，不破坏旧字段。
- `/api/data-statistics/trends` 支持新增趋势字段。
- RootAuth 权限不变。
- 默认查询近 7 天。
- 180 天限制仍然生效。
- 小时粒度有更严格限制。
- 没有新增写入主业务表的逻辑。
- 不影响计费、充值、日志写入主流程。

### 前端验收

- 数据为空时展示空状态，不报错。
- 图表在移动端不溢出。
- 长模型名、渠道名不撑破布局。
- 切换时间范围、模型、分组、渠道后图表和表格同步刷新。
- 环比为 `null` 时显示“新增”或“不适用”，不显示 `Infinity`。
- 金额、额度、百分比、耗时格式统一。

### 安全验收

- 页面不展示 prompt 原文。
- 页面不展示 API Key 或令牌明文。
- 页面不提供余额修改入口。
- 页面不提供渠道禁用入口。
- 负数 quota 和异常余额只做只读提示。
- 聚合计算避免整数溢出。

## 8. 默认决策

- 不新开页面，继续增强 `/data-statistics`。
- 第一版不新增图表依赖，使用现有 `recharts`。
- 第一版不新增快照表，继续实时聚合。
- 第一版保留趋势明细表，但把图表作为主展示。
- 第一版只给 Root 管理员使用。
- 第一版不做导出，导出能力后续单独规划。
- 第一版不在统计页做任何处置动作。
- 第一版优先覆盖用户、充值、消耗、请求、Token、错误、耗时、模型、分组、渠道。

## 9. 推荐开工顺序

1. 后端补齐 `summary` 的总用户和上一周期对比。
2. 后端补齐 `trends` 的累计总用户和充值次数。
3. 前端拆分当前大组件。
4. 前端新增用户趋势、收入趋势、消耗趋势、质量趋势。
5. 前端总览卡片显示环比。
6. 后端补齐错误模型/错误渠道排行。
7. 前端新增结构分析和异常观察。
8. 做一次生产数据只读验收。

这一顺序可以把风险压低：先做只读扩展，再做图表展示，不触碰充值、计费、渠道选择、请求转发等核心链路。
