# 分时段计费独立页面改造规划

## 1. 目标定位

这里讨论的不是“限时特价”，而是系统里的“分时段计费”能力升级。

当前系统已经具备通过 `tiered_expr` 表达式实现时间倍率的能力，例如夜间半价、周末八折。但这个能力藏在模型定价的高级表达式里，配置门槛高，也不适合运营日常维护。

新的目标是做一个独立的“分时段计费”页面，让管理员用活动规则的方式配置：

- 哪些分组享受折扣。
- 哪些模型享受折扣。
- 哪个分组下的哪个模型享受折扣。
- 从几号到几号生效。
- 每天几点到几点生效。
- 当前和未来优惠如何展示给用户。

第一版重点解决“好配置、好预览、好验收”，不要求管理员深入模型表达式编辑器。

## 2. 当前功能基础

当前分时段计费能力主要来自模型分层计费表达式。

相关后端：

- `setting/billing_setting/tiered_billing.go`
- `pkg/billingexpr/run.go`
- `pkg/billingexpr/settle.go`
- `relay/helper/price.go`
- `relay/helper/billing_expr_request.go`
- `service/tiered_settle.go`
- `service/log_info_generate.go`

相关前端：

- `web/default/src/features/system-settings/models/model-pricing-sheet.tsx`
- `web/default/src/features/system-settings/models/tiered-pricing-editor.tsx`
- `web/default/src/features/pricing/lib/billing-expr.ts`
- `web/default/src/features/pricing/lib/tier-expr.ts`

当前表达式支持这些时间函数：

```text
hour("Asia/Shanghai")     当前小时，0-23
minute("Asia/Shanghai")   当前分钟，0-59
weekday("Asia/Shanghai")  星期，0=周日，6=周六
month("Asia/Shanghai")    月份，1-12
day("Asia/Shanghai")      日期，1-31
```

这说明底层已经能做“每天某时段倍率”，但还缺一个适合运营使用的规则配置层。

## 3. 当前问题

### 3.1 入口太深

管理员现在要先进入模型定价，再切到分层计费或表达式编辑，才能配置时间条件。实际运营只想做“某分组夜间半价”，不应该被迫理解表达式。

### 3.2 只能围绕单模型配置

当前更像“给某个模型写一段计费表达式”。如果想配置：

- VIP 分组所有模型夜间八折。
- 某个模型全站半价。
- 某个分组的某个模型周末半价。

就会变成到处改模型配置，维护成本高，也容易漏模型。

### 3.3 用户侧没有促销预告

用户不知道哪些模型什么时候更便宜，也不知道自己所在分组是否享受优惠。运营活动不能只在后台生效，还需要在用户侧有可理解的展示。

### 3.4 叠加逻辑容易误解

当前表达式里的多条规则通常是相乘。例如夜间半价 `0.5` 和周末八折 `0.8` 同时命中，最终是 `0.4`。

独立促销规则如果继续默认叠加，容易出现配置漏洞和非预期超低价。第一版建议默认不叠加。

## 4. 产品方案

### 4.1 新增独立管理页

建议新增页面：

```text
管理员 -> 分时段计费
```

建议路由：

```text
/time-pricing
```

也可以放在：

```text
系统设置 -> 计费设置 -> 分时段计费
```

但从使用体验看，分时段计费更像运营活动管理，不建议继续藏进模型定价编辑弹窗里。

### 4.2 页面结构

新页面建议分为 5 个区域：

- 顶部概览：当前生效规则数、即将开始规则数、今日最低折扣、配置异常提示。
- 规则列表：启用状态、名称、作用范围、日期范围、每日时段、折扣、当前状态。
- 新建/编辑规则抽屉：配置作用范围、时间、折扣、优先级和展示文案。
- 生效预览：选择用户分组和模型，模拟当前/指定时间的最终倍率。
- 用户侧预告设置：控制是否在模型广场、定价页、仪表盘展示优惠。

### 4.3 规则配置项

每条规则建议包含：

```json
{
  "id": "night-vip-half",
  "name": "VIP 夜间半价",
  "enabled": true,
  "timezone": "Asia/Shanghai",
  "start_date": "2026-07-10",
  "end_date": "2026-07-31",
  "daily_start_time": "21:00",
  "daily_end_time": "06:00",
  "days_of_week": [0, 1, 2, 3, 4, 5, 6],
  "scope_type": "group_model",
  "groups": ["vip"],
  "models": ["deepseek-v4-flash"],
  "multiplier": 0.5,
  "priority": 100,
  "stacking": "exclusive",
  "user_visible": true,
  "user_title": "VIP 夜间半价",
  "user_description": "每天 21:00 至次日 06:00，deepseek-v4-flash 享 5 折计费"
}
```

字段说明：

- `timezone`：默认 `Asia/Shanghai`。
- `start_date/end_date`：活动总日期范围。
- `daily_start_time/daily_end_time`：每天的生效时段，允许跨天，例如 `21:00 -> 06:00`。
- `days_of_week`：可选，默认每天。
- `scope_type`：作用范围。
- `multiplier`：计费倍率，`0.5` 表示五折，`0.8` 表示八折。
- `priority`：规则优先级，用于多规则冲突处理。
- `stacking`：第一版建议固定为 `exclusive`，即不叠加。
- `user_visible`：是否对用户展示促销预告。

### 4.4 作用范围

作用范围建议提供 4 种模式：

| 模式 | 说明 |
| --- | --- |
| 全部 | 所有分组、所有模型 |
| 按分组 | 指定分组下全部模型 |
| 按模型 | 所有分组下指定模型 |
| 按分组 + 模型 | 指定分组下指定模型 |

第一版不建议支持手输模型名。模型选择应复用“已启用模型”下拉数据。

分组选择应复用当前分组配置，不允许输入不存在的分组。

### 4.5 时间范围

需要同时支持两层时间：

1. 活动日期范围：从几号到几号。
2. 每日生效时段：每天几点到几点。

示例：

```text
2026-07-10 到 2026-07-31
每天 21:00 到次日 06:00
VIP 分组 deepseek-v4-flash 五折
```

如果只设置日期范围、不设置每日时段，表示日期范围内全天生效。

如果每日时段跨天，例如 `21:00 -> 06:00`，系统预览必须明确显示“跨天”。

### 4.6 折扣配置

前端建议以“折扣”方式展示，后端存储倍率。

示例：

| 前端显示 | 后端倍率 |
| --- | --- |
| 5 折 | `0.5` |
| 8 折 | `0.8` |
| 原价 | `1` |

第一版建议限制：

- 最小倍率：`0.01`
- 最大倍率：`1`
- 精度：最多 4 位小数

先只做优惠，不做加价。加价容易引发用户争议，后续确实需要再单独开开关。

### 4.7 冲突处理

第一版建议默认规则不叠加，避免配置漏洞。

当多个规则同时命中时，按以下顺序选中一条：

1. `priority` 更高的规则优先。
2. 优先级相同，作用范围更具体的优先：`分组+模型 > 模型 > 分组 > 全部`。
3. 仍相同，倍率更低的优先。
4. 仍相同，创建时间更早的优先。

页面预览要显示：

```text
当前命中：VIP 夜间半价
被覆盖规则：全站夜间八折
最终倍率：0.5
```

后续如果要支持叠加，可以新增 `stacking = multiply`，但不建议第一版开放。

## 5. 用户侧促销预告

### 5.1 展示位置

用户侧建议展示在：

- 模型广场 / 模型定价页：模型卡片或价格表里显示优惠标签。
- 控制台概览：展示“当前可用优惠”和“即将开始优惠”。
- 钱包 / 充值页不展示，避免和充值优惠混淆。

### 5.2 展示内容

对用户展示时必须脱敏和按权限过滤：

- 只展示用户所在分组可享受的规则。
- 只展示用户实际可用的模型。
- 不展示后台规则 ID、内部优先级、其他分组名称。

示例文案：

```text
deepseek-v4-flash
VIP 夜间半价
每天 21:00 - 06:00，当前 5 折计费
```

即将开始：

```text
周末八折
7 月 13 日 00:00 开始，周六/周日全天生效
```

### 5.3 用户侧状态

用户侧可以分 3 种状态：

- 生效中：当前请求会按优惠倍率计费。
- 即将开始：未来 7 天内会生效。
- 已结束：默认不展示，除非后续做活动历史。

### 5.4 价格说明

用户侧必须明确“按实际请求计费时自动应用”，不要让用户误以为充值金额打折。

建议文案：

```text
该优惠作用于请求计费，不影响充值金额。
```

## 6. 后端实现规划

### 6.1 推荐第一版存储方式

为了减少迁移和上游冲突，第一版可以先走 `options` 表保存 JSON 配置：

```text
TimePricingRules
```

配置规模通常不会很大，读取后可做内存缓存。后续如果规则很多、需要审计历史或复杂查询，再迁移到独立表。

如果直接上表，建议表名：

```text
time_pricing_rules
```

但第一版不是必须。

### 6.2 后端配置结构

建议新增配置结构：

```json
{
  "enabled": true,
  "rules": [],
  "user_notice_enabled": true,
  "preview_days": 7
}
```

规则列表中的模型、分组保存为精确名称数组。

### 6.3 结算接入点

计费链路建议保持最小改动：

- 不改渠道选择。
- 不改模型映射。
- 不改 token 统计。
- 不改基础模型价格。
- 在基础扣费结果算出后，应用“分时段计费倍率”。

推荐接入位置：

1. 先按现有逻辑计算基础 quota。
2. 使用 `OriginModelName` 或实际计费模型名做规则匹配，需要在规划落地前统一口径。
3. 根据用户分组、模型名、当前时间找命中规则。
4. 用命中倍率调整最终扣费。
5. 日志记录命中规则快照。

模型口径建议：

- 展示和配置使用用户看到的模型名。
- 计费匹配优先使用 `relayInfo.OriginModelName`。
- 如果某些渠道依赖映射后模型名，后续可增加“按映射后模型匹配”的高级开关，但第一版不做。

### 6.4 命中计算

后端应提供统一函数：

```text
ResolveTimePricingRule(group, model, timestamp) -> matched_rule, multiplier
```

它只做纯规则判断，不访问请求内容，不读取 prompt。

判断顺序：

1. 总开关开启。
2. 规则启用。
3. 当前日期在 `start_date/end_date` 内。
4. 当前星期在 `days_of_week` 内。
5. 当前时间在 `daily_start_time/daily_end_time` 内。
6. 用户分组和模型命中作用范围。
7. 多规则按冲突策略选出一条。

### 6.5 日志记录

请求日志建议在 `Other` 中增加轻量快照：

```json
{
  "time_pricing": {
    "matched": true,
    "rule_id": "night-vip-half",
    "rule_name": "VIP 夜间半价",
    "scope_type": "group_model",
    "multiplier": 0.5,
    "timezone": "Asia/Shanghai"
  }
}
```

不要把完整规则配置、其他分组、其他模型列表写入请求日志。

这样管理员后续看用量日志时能解释“为什么这次请求更便宜”。

### 6.6 管理端接口

建议新增：

```text
GET  /api/time-pricing/settings
PUT  /api/time-pricing/settings
POST /api/time-pricing/preview
GET  /api/time-pricing/enabled-models
GET  /api/time-pricing/groups
```

权限：

- 管理配置接口：`RootAuth`。
- 只读预览接口：`RootAuth` 或 `AdminAuth`，看后台权限设计。
- 用户侧预告接口：`UserAuth`。

用户侧：

```text
GET /api/user/time-pricing/promotions
```

只返回当前用户可见、可用、未结束的促销。

### 6.7 缓存

分时段计费处于请求热路径，不能每次请求都查数据库。

建议：

- 配置更新时刷新内存缓存。
- 运行时从内存读取规则。
- 多实例部署时可用 Redis 发布/订阅或沿用现有 options 刷新机制。
- 预告接口可以做 30-60 秒短缓存。

### 6.8 安全约束

必须做：

- 倍率必须大于 0。
- 第一版倍率最大不超过 1。
- 日期范围必须合法，结束时间不能早于开始日期。
- 规则数量设置上限，例如最多 200 条。
- 单条规则的分组和模型数量设置上限，例如各最多 200 个。
- 用户侧接口必须按当前用户分组过滤。
- 日志不记录 prompt、请求体、token key、完整内部配置。


```text

```

侧边栏位置：

```text
管理员 -> 分时段计费
```

### 7.2 规则表单交互

规则表单建议是抽屉，不做大弹窗。

字段：

- 规则名称。
- 启用开关。
- 活动日期范围。
- 每日生效时段。
- 星期选择。
- 作用范围模式。
- 分组多选。
- 模型多选。
- 折扣输入。
- 用户侧是否展示。
- 用户侧标题和说明。
- 优先级。

模型多选必须来自已启用模型接口。

分组多选必须来自分组配置。

### 7.3 预设模板

内置几个按钮，只填表不保存：

- 夜间半价：每天 21:00-06:00，5 折。
- 凌晨低峰 7 折：每天 00:00-08:00，7 折。
- 周末八折：周六、周日全天，8 折。
- 月末促销：选择日期范围后全天 8 折。

### 7.4 预览能力

管理端必须有“模拟预览”：

- 选择分组。
- 选择模型。
- 选择时间。
- 展示命中规则、最终倍率、被覆盖规则。

这是防止配置漏洞的关键。

### 7.5 用户侧展示

模型广场或定价页显示：

- 当前生效优惠标签。
- 即将开始优惠标签。
- 优惠时间段。
- 折后倍率。

控制台概览可以增加轻量提示：

```text
当前可用优惠：VIP 夜间半价，21:00 - 06:00 生效。
```

用户侧不要显示后台规则列表，不要显示其他分组。

## 8. 与现有表达式计费的关系

第一版建议不要把新页面规则强行写回每个模型的 `tiered_expr`。

原因：

- 新页面要支持按分组配置，而 `tiered_expr` 当前主要按模型保存。
- 如果写回每个模型表达式，后续同步上游模型价格会更难合并。
- 用户侧预告和规则审计需要独立元数据，表达式不适合作为唯一数据源。

更稳的方式是：

- 现有模型价格继续负责“基础价格”。
- 分时段计费页面负责“运营折扣倍率”。
- 结算时在基础价格结果上叠加一个独立折扣层。

后续如果要把两者合并，可以做“编译为表达式”的高级模式，但不作为第一版目标。

## 9. 计费公式建议

建议第一版公式：

```text
最终扣费 = 现有计费结果 * 分时段倍率
```

其中“现有计费结果”已经包含：

- 模型倍率或模型价格。
- 分组倍率。
- 分层计费表达式。
- 缓存、图片、音频等已有计费逻辑。

这样改动最少，也最容易解释：

```text
先按原价算，再按活动打折。
```

需要在日志里记录：

- 原始扣费。
- 分时段倍率。
- 最终扣费。

## 10. 分阶段实施建议

### 第一阶段：独立配置 + 热路径折扣

目标：

- 新增独立管理页。
- 支持分组、模型、分组+模型范围。
- 支持日期范围和每日时段。
- 支持用户侧促销预告。
- 请求计费时应用倍率。
- 日志记录命中规则。

不做：

- 复杂叠加。
- 节假日日历。
- 加价规则。
- 每用户单独促销。
- 充值折扣联动。

### 第二阶段：增强运营能力

可增加：

- 规则复制。
- 批量启停。
- 规则命中统计。
- 预计影响收入。
- 活动日历视图。
- 用户侧更多展示位。

### 第三阶段：高级计费融合

可考虑：

- 与模型表达式编辑器互相引用。
- 按渠道/供应商做活动。
- 按订阅等级做活动。
- 支持节假日模板。
- 支持叠加策略：不叠加、相乘、取最低价、按优先级。

## 11. 验收标准

第一版验收建议：

- 能新建“VIP 分组每天 21:00-06:00 全模型五折”。
- 能新建“某个模型全站周末八折”。
- 能新建“某个分组的某个模型在指定日期范围内半价”。
- 模型选择来自已启用模型下拉，不能手输。
- 分组选择来自现有分组配置，不能手输不存在分组。
- 跨天时间段预览正确。
- 多规则冲突时只命中一条，并说明覆盖关系。
- 用户侧只能看到自己分组可享受的促销。
- 请求日志能看到命中规则和倍率。
- 关闭总开关后，不影响现有模型计费。
- 未命中规则时，扣费完全等同当前逻辑。

## 12. 推荐默认决定

建议最终按下面方式落地：
